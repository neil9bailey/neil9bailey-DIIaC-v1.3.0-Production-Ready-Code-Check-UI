# DIIaC — Key Vault Transition Guide

> **Status: COMPLETE — Key Vault `kv-diiac-vendorlogic` is provisioned and active.**
> Azure Key Vault is now the authoritative source for all DIIaC secrets.
> Run `bash scripts/pull-keyvault-secrets.sh` (or the PowerShell equivalent) before starting the stack.

---

## What changes between pre-KV and post-KV

| Area | Pre-Key Vault (now) | Post-Key Vault (end state) |
|------|--------------------|-----------------------------|
| **How secrets reach the app** | You create `.env` manually | `pull-keyvault-secrets.sh/.ps1` writes `.env` automatically |
| **Where secrets are stored** | Your local machine only | Azure Key Vault (encrypted, access-controlled, audited) |
| **Secret rotation** | Edit `.env` manually, restart containers | Re-run pull script, restart containers |
| **Azure CLI required** | No | Yes (`az login` + Key Vault Secrets User role) |
| **Signing key** | Generated locally (see below) | Pulled from Key Vault secret `diiac-signing-private-key-pem` |
| **Admin token** | You set any value in `.env` | Pulled from Key Vault secret `diiac-admin-api-token` |
| **OpenAI API key** | You paste into `.env` | Pulled from Key Vault secret `diiac-openai-api-key` |

## Phase A — Manual `.env` (fallback / offline only)

### 1. Create your `.env` file

```bash
cp .env.example .env
```

### 2. Generate a signing key locally

```bash
# Linux / Mac / Git Bash
mkdir -p .secrets
openssl genpkey -algorithm ed25519 -out .secrets/signing_key.pem
```

```powershell
# Windows PowerShell
New-Item -ItemType Directory -Force -Path .secrets | Out-Null
# Requires OpenSSL — install via: winget install ShiningLight.OpenSSL
openssl genpkey -algorithm ed25519 -out .secrets\signing_key.pem
```

Verify it worked:

```bash
head -1 .secrets/signing_key.pem
# Expected: -----BEGIN PRIVATE KEY-----
```

### 3. Fill in `.env`

Open `.env` in any text editor and set these values:

```dotenv
# ── Required ──────────────────────────────────────────────────────────────────

# Admin API token — any strong random string (min 32 chars).
# Generate one with: openssl rand -hex 32
ADMIN_API_TOKEN=<paste-generated-token-here>

# OpenAI key — needed for LLM-assisted compile features.
# Get from: https://platform.openai.com/api-keys
OPENAI_API_KEY=sk-...

# Signing key is loaded from the file you generated above.
# docker-compose.yml bind-mounts .secrets/signing_key.pem automatically.
# Leave SIGNING_PRIVATE_KEY_PEM blank — the file mount takes precedence.

# ── Optional (change only if ports conflict) ──────────────────────────────────
RUNTIME_HOST_PORT=8000
BRIDGE_HOST_PORT=3001
FRONTEND_HOST_PORT=5173
```

### 4. Start the stack

```bash
docker compose up
```

That's it. No Azure access needed at this stage.

## Phase B — Key Vault enabled (ACTIVE — use this)

### Prerequisites

- Azure Key Vault provisioned: `kv-diiac-vendorlogic` ✓ DONE
- Secrets populated (see `customer-config/vendorlogic/keyvault-secrets-manifest.md`)
- Your account has the `Key Vault Secrets User` role on the vault
- Azure CLI installed and logged in: `az login`

### Secrets that must exist in Key Vault

| Secret name | Description |
|-------------|-------------|
| `diiac-admin-api-token` | 64-char hex admin token |
| `diiac-signing-private-key-pem` | Ed25519 private key (PKCS#8 PEM, no newlines) |
| `diiac-openai-api-key` | OpenAI API key (`sk-...`) |

See `customer-config/vendorlogic/keyvault-secrets-manifest.md` for full detail.

### Switch to Key Vault

```bash
# Linux / Mac / Git Bash
bash scripts/pull-keyvault-secrets.sh

# Windows PowerShell
.\scripts\pull-keyvault-secrets.ps1
```

The script:
1. Connects to `kv-diiac-vendorlogic` using your `az login` session
2. Writes non-PEM secrets to `.env`
3. Writes `diiac-signing-private-key-pem` to `.secrets/signing_key.pem`

Then start the stack as normal:

```bash
docker compose up
```

### After Key Vault is live — what to delete

Once Key Vault is confirmed working:

- Delete the manually created `.env` (the pull script regenerates it)
- Delete the locally generated `.secrets/signing_key.pem` (the pull script regenerates it)
- Remove `ADMIN_API_TOKEN` and `OPENAI_API_KEY` from any notes — they now live in Key Vault only

## Integration architecture (no code changes required)

`app.py` and `backend-ui-bridge/server.js` always read from environment variables regardless of how they were sourced. The Key Vault integration is entirely in the pre-flight pull scripts — no runtime or bridge code changes are needed.

Secret flow:
```
Azure Key Vault (kv-diiac-vendorlogic)
        │
        ▼  scripts/pull-keyvault-secrets.sh
   .env + .secrets/signing_key.pem
        │
        ▼  docker compose -f docker-compose.yml -f docker-compose.staging.yml
governance-runtime + backend-ui-bridge + frontend
```
